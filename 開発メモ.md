# 開発メモ — NogiPhotoManager46

弓木奈於の生写真コレクション（約390セット・約1300枚）の所有管理アプリ。設計の決定内容と理由の記録。
（実装順は `task.md` を参照）

## 1. なぜ作り直すのか（旧iOS版の反省）
- 一番難しい「複数枚の自動切り抜き(Vision矩形検出)」から着手して不安定なまま消耗した。
- スキャナーとDBが未接続、写真一覧画面が未実装、カタログ未投入で全手入力前提だった。
- 教訓: **まず「持ってる/持ってない」を管理できる道具を完成させ、画像は後から足す。**

## 2. 主要な決定（ADR）

| # | 決定 | 理由 / 却下した案 |
|---|---|---|
| 1 | **Webアプリ(PWA)に転換**（Vite+React+TS+Tailwind+Dexie+vite-plugin-pwa、Vercel） | 無料Apple IDの署名7日失効が日常利用に致命的。Vercel+GitHubは運用経験あり・無料。ホーム画面追加でアプリ同然に使える。／却下: iOS継続（署名問題）、有料Developer Program（費用） |
| 2 | 所有粒度は**写真1枚ごと** | 実際の集め方（個別交換・部分所有）に一致。セットは全枚数所有で自動コンプ。 |
| 3 | 階層: Member > Binder > Year > Set > Photo。バインダー5冊: 2020&2021 / 2022&2023 / 2024&2025 / **2026&2027(新設)** / 封入(年層なし) | 物理バインダーと1対1。封入は year=null で年層スキップ。 |
| 4 | テンプレ(standard3/five5/rareSet8/event6/single1)は**初期値生成専用**。保存するのは展開後の写真枠のみ | 旧isFivePoses(bool)は8種等を表現できない。生成後の枠増減・改名を自由にするため、テンプレとの整合性管理を持たない。 |
| 5 | **カタログ(静的JSON)と所有(IndexedDB)を完全分離** | カタログ更新が所有○×を壊す事故が構造的に起きない。同期ロジックが激減。カタログ更新=JSON編集→push→自動デプロイ。 |
| 6 | stableIDは**機械的採番** `<member>:<setCode>:<slot>`（例 `yumiki_nao:2022-03a:yori`）。凍結・変更禁止 | 名前由来IDは表記修正でID変更したくなり照合が壊れる。IDは照合キーであって表示物ではない。 |
| 7 | カタログから消えたセットは**削除しない**（追加・更新のみ） | 所有データ保護の最強手段。自分がカタログ管理者なので削除が必要な場面がほぼない。／却下: tombstone+引き継ぎID（個人運用には過剰） |
| 8 | メンバー選択画面なし。前回選択を記憶+ヘッダーメニュー切替 | 1人しかいないのに毎回選択は無駄。メンバー追加時も画面改変ゼロ。 |
| 9 | **エクスポート/インポートはP1必須** | 所有データはこの端末にしかない。iOSのPWAはホーム画面から削除するとIndexedDBも消える。バックアップJSONが生命線。 |
| 10 | Undo機構に依存しない | 単品トグルは再タップで戻る。破壊的な一括操作のみ確認ダイアログ。 |
| 11 | ネットからの画像自動収集はしない | 生写真画像は乃木坂/ソニーの著作物。自分の所有物を自分で撮影/スキャンして添付（P1.5以降）。 |

## 3. 初期データ変換ルール（✓/◦リスト → JSON）
- `✓` = 全ポーズ所有 ／ `◦`(注記なし) = 全未所有
- **`◦` + 末尾注記（「ヨリチュウ」等）= 注記の通常ポーズが未所有（欠け）、残りの通常ポーズは所有**（例: 君に叱られた ◦「ヨリヒキ」→ チュウのみ所有）
- **8種(rareSet8)のR/SRは ✓ でも自動所有にしない**。理由: 元リストでは「通常3種を持っていれば✓」を付け、R/SRの所有は別管理（線引き）していたが、その情報がメモ添付時に失われた。よってR/SR所有はリストから判断せず、`ownership-overrides.txt`（非公開・gitignore）で個別指定する。完コンプ時は本人が別途連絡
- `〜〜〜` = 物理ページ区切り → `pageBreakAfter: true`
- `（8種類）（通常3種+R1種+SR4種）` = rareSet8（ヨリ/チュウ/ヒキ + R + SR①〜④）
- **矛盾（✓なのに注記あり、枚数不明等）は変換スクリプトが「要確認レポート」に出力し、手動で裁定してから確定**
- **GitHubリポジトリはPublicのため、公開カタログJSONに `owned`（所有）を含めない**。所有初期値は非公開の所有ファイルに分離し、P1-5のインポートで一度だけ取り込む
- 変換スクリプトはリポジトリに保存。原文リスト(✓付き)と所有ファイルは非公開(gitignore: `*.ownership.json` / `catalog-source*.txt`)にして再生成可能にする

## 4. カタログJSONスキーマ（要点）
```json
{
  "schemaVersion": 1,
  "catalogVersion": 1,
  "member": { "id": "yumiki_nao", "name": "弓木奈於" },
  "binders": [ { "id": "b2020-2021", "name": "弓木2020&2021", "sortIndex": 0, "sealed": false } ],
  "sets": [ {
    "id": "2022-03a", "binderId": "b2022-2023", "year": 2022,
    "name": "スペシャル衣装28", "sortIndex": 1410,
    "template": "standard3", "owned": ["yori", "hiki"],
    "note": "ヨリチュウ", "pageBreakAfter": false
  } ]
}
```
- `owned` は初期seed専用のoptional（初回投入時のみ解釈。以後の所有はIndexedDBが正）
- `template` 指定で枠自動展開。変則セットは `photos:[{slot,label,rarity,owned}]` を明示（SRラベル差し替え等）
- setCode採番: 月別=`2022-03a`、MV=`mv-ohitorisama`、卒コンT=`grad-kubo-t`、封入=`sealed-sg34`、手動追加=`user-<uuid>`

## 5. 画面構成（3タブ・モバイルファースト）
1. **コレクション**: バインダー一覧（全体コンプ率+進捗バー）→ 年 → セット一覧（セグメント「すべて/未コンプ/コンプ」+検索）→ セット詳細（ポーズカードのグリッド、タップでトグル、下部に「すべて所有」）。single1は一覧から直接トグル。封入は年層スキップ。
2. **統計**: 総所有/コンプ率、バインダー別・年別・レアリティ別
3. **設定**: エクスポート/インポート、データ件数、メンバー管理
- ポーズカードは最初から生写真比率(89:127)の画像領域を確保 → P1.5でサムネ差し替えのみで拡張
- カードの右上チェックは常にトグル専用領域（P1.5でカード本体タップ=画像拡大に変わっても手癖が壊れない）

## 6. 主要リスクと対策
| リスク | 対策 |
|---|---|
| ホーム画面からPWA削除→IndexedDB全損 | エクスポートを設定画面の1等地に。`navigator.storage.persist()` 要求。定期バックアップを習慣化 |
| カタログJSONの手編集ミス（ID変更・重複） | 変換スクリプト経由＋検証スクリプト（ID重複・枠数・件数チェック）必須。ID凍結規約 |
| ✓リスト自体の転記ミス | 要確認レポート＋物理バインダー順ソートで実物と突き合わせやすくする |
| PWAキャッシュでカタログ更新が反映されない | カタログJSONはnetwork-first取得＋catalogVersion表示（設定画面） |
| 画像でストレージ肥大（P1.5） | 保存前に長辺1024px程度へ縮小。一覧はサムネ専用データのみ読む |
| 公開URLのアクセス | 所有データは端末ローカルなので漏れない。サイトはnoindex。スクショ公開時は画像添付セットを写さない |

## 7. 不採用と記録するもの（再提案防止）
